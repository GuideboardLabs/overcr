# OverCR v2.2.0 — Operator Interface Architecture

## Overview

The Operator Interface Layer adds a terminal-first, read-only observatory
to OverCR. It renders formatted views of canonical filesystem state for
operator awareness and decision support.

The TUI is an **observatory, not a cockpit**. It displays state; it does
not create or advance it. Approval actions are proposed, not executed.

## Design Decisions

### 1. Read-Only Rendering

All TUI views read from canonical filesystem state (task files, audit
log, memory index). They never write to runtime state directly. This
preserves the filesystem-first architecture where disk is truth.

### 2. Approval Actions as Data Objects

`ApprovalQueue.propose_approval()` and `propose_rejection()` return
`ApprovalAction` dataclass instances. These carry task_id, action,
rationale, and operator identity. The caller is responsible for routing
them through `OverCRRuntime.process_approval()` which enforces the
approval gate governance.

This prevents the TUI from becoming a governance bypass vector.

### 3. Deterministic Fallback

Every view provides both `render()` (rich-formatted) and `render_plain()`
(plain text) methods. The plain fallback produces identical semantic
content without rich markup, ensuring operability in:
- Constrained terminals without true-color support
- Log file capture and piping contexts
- Automated testing and comparison

### 4. Resilient to Partial Failure

Missing runtime components produce empty/degraded views, not crashes:
- No task files → "No tasks found"
- No audit log → "No entries matching filters"
- No memory index → Memory summary shows all zeros
- Workers unavailable → Health shows "degraded"

### 5. Rich Library

The TUI uses the `rich` Python library for terminal formatting. Rich is
a pure-Python library with no browser, web, or cloud dependencies. It
degrades gracefully on terminals without true-color support.

Textual is NOT used — it requires an interactive event loop and would tie
the TUI to a specific frontend runtime. The declarative `render()`
method approach keeps the TUI frontend-agnostic.

### 6. No Keybinding Dispatch

`KeyBindings` is a declarative data structure. It defines bindings as
data (key → action → description) but does not capture input or dispatch
events. Actual key handling depends on the frontend context (a Rich
console, a Textual app, or plain stdin reading).

## View Hierarchy

```
Dashboard
├── StatusBar              (runtime health, worker count, memory summary)
├── ApprovalQueue          (pending approvals, rationale, approve/reject paths)
├── TaskView               (task list, detail, state lifecycle, audit refs)
├── AuditView              (append-only stream, filtering, replay navigation)
├── WorkflowView           (DAG visualization, node states, approval gates)
├── PacketInspector        (request/response packets, L1-L6, provenance, routing)
└── Memory Summary         (active/stale/total counts from memory layer)
```

## Data Flow

```
Filesystem State                    TUI Views
──────────────                    ──────────
orchestration/tasks/*.json  ───→  TaskView, PacketInspector, ApprovalQueue
runtime/audit.jsonl          ───→  AuditView
memory/index.jsonl           ───→  StatusBar (memory summary)
subagents/                  ───→  StatusBar (worker availability)
config/                      ───→  (future: provider routing display)
```

All arrows point FROM filesystem TO views. No arrows point back.
The TUI never writes to canonical state.

## Approval Flow

```
Operator sees pending approval in ApprovalQueue
         │
         ├─→ propose_approval(task_id, reason) → ApprovalAction(action="approve")
         │                                              │
         │                                              └─→ Caller routes through
         │                                                   OverCRRuntime.process_approval()
         │                                                   + ApprovalGate.enforce_gate()
         │                                                   + AuditWriter.approval_action()
         │
         └─→ propose_rejection(task_id, reason) → ApprovalAction(action="reject")
                                                        │
                                                        └─→ Caller routes through
                                                             OverCRRuntime (sets state)
                                                             + AuditWriter.approval_action()
```

At no point does the TUI bypass the approval gate. The `ApprovalAction`
object is a proposal, not an execution.

## Widget Architecture

```
Widgets (reusable rendering primitives)
├── TableWidget             # Deterministic table (rich + plain fallback)
├── PanelWidget             # Bordered container (rich + plain fallback)
├── LogViewWidget           # Append-only stream renderer (audit, logs)
└── StatusBadge             # Colored status indicators (tasks, nodes, health)
```

Widgets are deterministic: same input always renders the same output.
They have no side effects and no state.
