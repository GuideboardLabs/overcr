## Checklist (30 items)

### Dashboard & Demo Output (AC-001 to AC-005)
- AC-001: Demo scripts are parseable and runnable
- AC-002: Dashboard distinguishes substrate from workload
- AC-003: No ambiguous or misleading operator output
- AC-004: Error messages are clear and actionable
- AC-005: Version information clearly displayed

### Approval Queue (AC-006 to AC-010)
- AC-006: Approval queue artifacts are well-formed JSON
- AC-007: Every pending approval shows operator_action_required: true
- AC-008: Approval artifacts include operator identity and chain
- AC-009: No auto-approval path exists for execution plans
- AC-010: Refusal paths are clear

### Packet Trace (AC-011 to AC-015)
- AC-011: Packet traces show full source→target routing
- AC-012: Validation levels (L1-L6) visible in traces
- AC-013: No direct subagent-to-subagent traces

### Audit Trail (AC-016 to AC-020)
- AC-016: Audit entries are timestamped and sequential
- AC-017: Every state transition produces an audit entry
- AC-018: Audit covers full lifecycle
- AC-019: Audit entries include actor identity
- AC-020: Tampered audits are detectable

### Release Validation (AC-021 to AC-024)
- AC-021: Release scripts run end-to-end
- AC-022: Release manifest is complete
- AC-023: Archive excludes runtime debris
- AC-024: SHA256 manifest verifiable

### Safety Boundaries (AC-025 to AC-030)
- AC-025: No bypass of approval gates
- AC-026: No direct shell execution path

- AC-028: Governance overrides rejected at validation

## Operator Notes

The operator acceptance module is designed for a human to review. Each check item includes:
- **status:** PASS / FAIL / WARN / SKIP
- **notes:** Why it passed or failed
- **category:** Which operational area it covers

Operators should focus on:
1. Any FAIL item (especially in the safety category)
2. Any WARN item that affects their operational workflow
3. The unsafe_paths_found list in the report
