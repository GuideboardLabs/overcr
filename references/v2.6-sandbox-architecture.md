# OverCR v2.6.0 — Controlled Execution Sandbox

## Release Summary

v2.6.0 introduces a governed execution sandbox for PypER. This is the most security-sensitive subsystem in OverCR. Every command passes through allowlist validation, shell metacharacter detection, network access blocking, and filesystem boundary enforcement — all before a single byte executes.

Execution authority is `operator-approved-sandbox-only`. The sandbox is local-first, `shell=False` everywhere, and impossible to silently escalate.

## Architecture Decisions

### 1. Allowlist, Not Denylist
`ALLOWED_COMMANDS` contains exactly 14 commands: ls, pwd, cat, head, tail, grep, find, echo, cp, mv, mkdir, rm, touch, diff. Everything not on this list is rejected. This is defense-in-depth — the denylist in `BLOCKED_COMMANDS` is supplementary, not the primary gate.

### 2. Shell=False, Always
Every subprocess call uses `shell=False` with an argv array. No string interpolation into a shell context. Shell metacharacters in arguments are detected and blocked before the subprocess boundary.

### 3. Layered Validation
`CommandPolicy.validate_request()` runs 9 checks: approval artifact, allowlist, shell metacharacters, chaining operators, redirects, path traversal, sandbox boundary, timeout range, and sandbox root existence. A single failure blocks execution. Nothing is advisory.

### 4. Pre-Execution Snapshots
Before any mutating command (cp, mv, mkdir, rm, touch, echo), `RollbackSnapshot` captures file content and metadata. Snapshots are append-only and preserved indefinitely. Every receipt lists its snapshot refs.

### 5. Default-Deny Network
`NetworkGuard` checks command names (curl, wget, ssh, etc.), argument patterns (URLs, IPs, domains with network context), and localhost access. There is no allowlist for network — all network is denied in v2.6.0.

### 6. Immutable Receipts
Every execution produces an `ExecutionReceipt` with: operator identity, approval chain, full argv, output hashes (SHA-256), timing, modified paths, snapshot refs, rollback state, and governance flags. Receipts are JSON on disk and never deleted.

### 7. Real Subprocess Execution
The sandbox runs allowed commands via `subprocess.run()`. This is REAL execution, not simulated. Tests use temp directories to isolate side effects.

## File Structure

```
sandbox/
├── __init__.py                      # Package exports
├── allowed_commands.py              # Canonical allowlist (14 commands)
├── command_policy.py                # 9-check validation engine
├── filesystem_guard.py              # Sandbox boundary enforcement
├── network_guard.py                 # Default-deny network access
├── rollback_snapshot.py             # Pre-exec snapshots + rollback
├── execution_receipt.py             # Immutable audit record
├── sandbox_runner.py                # Governed execution orchestrator
└── schema/
    ├── execution_request.schema.json
    └── execution_receipt.schema.json
```

## Security Boundaries

| Boundary | Mechanism | Enforcement |
|---|---|---|
| Command allowlist | `ALLOWED_COMMANDS` (14 commands) | Pre-exec validation |
| Shell injection | Metacharacter detection in argv | Pre-exec validation |
| Command chaining | `&&`, `||`, `;`, `\|` detection | Pre-exec validation |
| Path traversal | `../` and `..\\` detection | Pre-exec + realpath |
| Filesystem boundary | `realpath` prefix check against sandbox root | Pre-write check |
| Network access | Command name + URL/IP/domain detection | Pre-exec validation |
| Symlink escape | `realpath` resolution before write | Pre-write check |
| Timeout | `subprocess.run(timeout=...)` | Kernel-level SIGTERM |
| Audit integrity | Append-only JSON receipts | Filesystem persistence |

## What Was NOT Implemented (by Design)

- No sudo/doas/pkexec/su
- No package installation (apt, pip, npm, etc.)
- No shell interpreters (bash -c, sh -c, python -c)
- No network tools (curl, wget, ssh, nc)
- No Docker/container orchestration
- No kernel isolation (Firejail, Bubblewrap — future extension)
- No persistent agents or daemons
- No autonomous execution loops
- No self-modifying workflows
- No browser automation

## Future v2.7+ Extension Points

1. **Kernel Isolation** — Firejail/Bubblewrap profiles for filesystem and network jailing at the kernel level
2. **Seccomp Filters** — syscall allowlists per command type for defense-in-depth
3. **Extended Allowlist** — operator-configurable command additions with risk classification
4. **CryER Integration** — sandbox execution as output handler for CryER signal response workflows
5. **Diff-Based Rollback** — store diffs instead of full file snapshots for storage efficiency
6. **Resource Limits** — CPU and memory caps per execution
7. **TUI Integration** — live execution dashboard with pause/abort/rollback controls
