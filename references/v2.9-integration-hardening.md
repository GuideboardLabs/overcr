# OverCR v2.9.0 — Integration Hardening

## Purpose

v2.9.0 stabilizes the entire v2 architecture into a coherent, production-grade
substrate candidate by focusing on integration integrity, migration safety,
consistency validation, replay verification, packaging discipline, and
release hardening.

This release does NOT add major new capabilities. It proves the v2 stack
is internally coherent, replayable, portable, auditable, and recoverable.

## Architecture

```
integration/
├── schema_registry.py        Centralized schema discovery and verification
├── system_validator.py       Full system structure validation
├── replay_validator.py       Replay determinism and audit reconstruction
├── state_consistency.py      Cross-system reference integrity
├── release_integrity.py      Release cleanliness checks
├── migration_checker.py      v1 → v2 compatibility validation
├── compatibility_matrix.py   Machine-readable compat report
└── recovery_verifier.py      Cold-start and recovery simulation
```

## Validation Philosophy

Every validator follows these rules:

1. **Report only** — never mutate state
2. **Deterministic** — same input = same output
3. **Filesystem-first** — all validation reads from disk
4. **No external services** — all checks are offline
5. **No silent repair** — validators detect, never auto-fix

## System Validator Coverage

- Required directories (runtime, memory, tui, workflow_library, etc.)
- Required files (validate_packet.py, test_manifest.json, run_all.py)
- Schema completeness and referential integrity
- Workflow template validation (all templates)
- Sandbox backend compatibility (local, bubblewrap, firejail)
- Audit log structure and trace file integrity
- Package version consistency across subsystems
- Replay prerequisites (manifest recovery tests, context replay flag)
- Frozen workflow immutability (all 5 templates)

## Replay Validation Coverage

- Determinism: same workflow + same input = same node order
- Audit structure: entry types preserved across runs
- Branch trace consistency (ConditionEvaluator determinism)
- State machine transition validity
- Receipt serialization round-trip

## State Consistency Coverage

- Task references (valid subagents, states)
- Workflow lineage (edge-to-node consistency, orphan detection)
- Memory references (parseable records)
- Provenance references
- Snapshot references
- Subworkflow lineage
- Receipt linkage (round-trip verification)
- Escalation lineage (record creation and resolution)

## Release Integrity Coverage

- pycache and .pyc leakage
- Transient runtime debris
- Orphaned receipts and stale snapshots
- Documentation completeness
- Version consistency
- Mutable artifacts (.gitignore, large files)
- Broken schema references

## Migration Checking

- V1 component preservation (validate_packet, subagents, task counter)
- Schema versioning consistency
- Workflow version pinning
- Backward compatibility metadata
- V2.3→v2.8 upgrade chain integrity

## Recovery Verification

All recovery scenarios use temporary directories — production state
is never mutated:

1. **Cold-start reconstruction**: fresh checkout, no runtime state
2. **Replay recovery**: reconstruct state from audit traces
3. **Partial artifact loss**: detect missing traces, load remaining
4. **Audit reconstruction**: group by run_id, verify timeline
5. **Workflow continuation**: deterministic re-execution
6. **Rollback restoration**: snapshot save/load and audit tracing
