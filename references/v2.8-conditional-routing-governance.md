# OverCR v2.8.0 — Conditional Routing Governance

## Routing Decision Lifecycle

Every routing decision follows this lifecycle:
1. **Evaluate** — `ConditionEvaluator` checks conditions against current state
2. **Match** — `RoutingPolicy` picks highest-priority matching edge
3. **Record** — `BranchTrace` captures the decision
4. **Transition** — `WorkflowStateMachine` validates and executes

## Routing Guarantees

1. **Deterministic**: Same state + same conditions = same routing decision
2. **Auditable**: Every decision has timestamp, operator, conditions evaluated
3. **Replayable**: BranchTrace reproduces the exact path taken
4. **Bounded**: Retries capped at max_retries (0-10)
5. **Visible**: All routing metadata embedded in workflow audit trace
6. **Operator-Aware**: Escalations route to operator review queues

## Remaining Workflow Risks (Honest Assessment)

1. **Condition blind spots**: Conditions are data-driven but limited to the fields captured in state snapshots. Information NOT in the snapshot cannot inform routing.

2. **Subworkflow timeout independence**: Subworkflow timeouts are independent of parent. A parent with 300s timeout could have a child that times out at 30s — the parent's routing must handle this.

3. **Priority inversion**: Conditional edges with equal priority may produce non-deterministic routing if multiple conditions match. Mitigation: ties are broken by insertion order (stable).

4. **Retry delay not enforced**: `retry_delay_s` is advisory. The caller must implement the sleep. This prevents the composition layer from blocking but places burden on the caller.

5. **Branch trace completeness**: The trace captures routing decisions but not WHY a condition evaluated a certain way beyond the condition dict itself. The raw evaluator output is preserved as metadata.
