# OverCR v2.3.0 — Workflow Governance

## Purpose

This document defines the governance rules that every workflow template and execution must satisfy. These rules are encoded in the software (validation, policy checks, approval gates) — this document serves as the human-readable specification.

## Governance Layers

### Layer 1: Template Schema Validation
Every template must pass JSON schema validation before registration.
- All required fields present
- All field types correct
- No duplicate node_ids or edge_ids
- No missing references in edges

### Layer 2: Node Validation
Every node in a template must:
- Reference a valid subagent (cryer, pyper, coder, knower)
- Declare a packet_type that belongs to its subagent
- Have a unique node_id
- Not exceed declared timeouts

### Layer 3: Edge Validation
Every edge must:
- Reference source and target nodes that exist
- Not create self-loops (source == target)
- Not create invalid handoff paths between subagents
- Declare accepted packet types

### Layer 4: DAG Integrity
The workflow graph must:
- Have at least one node
- Contain no cycles (Kahn's algorithm verification)
- Have all nodes reachable from a root

### Layer 5: Sovereignty
No subagent may route directly to another subagent.
- All cross-subagent edges must go through a valid OverCR handoff path
- The handoff path mapping is maintained in workflow_graph.py
- Unauthorized paths are rejected at template validation time

### Layer 6: Approval Gates
Approval points are mandatory, not advisory.
- Nodes in `approval_points` cannot execute without operator approval
- The executor pauses the workflow at approval points
- Only an explicit approve() call advances past the gate
- Rejections stop the workflow and record the reason

## Runtime Enforcement

### What workflows MUST do:
- Execute nodes in topological order
- Validate every node's output through L1-L6 validator
- Record every state transition in audit trace
- Respect declared timeout_seconds
- Activate deterministic fallback only when policy allows

### What workflows MUST NOT do:
- Invoke shell execution directly
- Mutate filesystem state without approval
- Spawn sub-workflows recursively
- Bypass approval_required fields
- Hide failed nodes from audit
- Discard or truncate audit entries
- Contact external services
- Execute browser automation
- Create autonomous loops
- Modify template definitions at runtime

## Audit Integrity

Every audit trace entry is:
- Immutable (append-only, JSONL format)
- Timestamped (UTC ISO 8601)
- Tagged with run_id and workflow_id
- Written to disk immediately (no in-memory-only buffering)

Traces survive across sessions. Replay reconstructs execution state from traces without re-executing any nodes.

## Operator Requirements

The operator:
- Is identified in every approval record
- Must explicitly approve before any gated node executes
- Can reject with a reason that is recorded
- Is responsible for all workflow outcomes

In automated/headless deterministic mode, approvals are auto-granted with a "Deterministic mode" annotation. This is acceptable for testing and demonstration, but production workflows should always require human operator intervention at approval points.

## Version Compatibility

- Template version 2.3.0 is the current version
- Future versions must maintain backward compatibility
- Template version mismatch during replay produces a warning, not an error
- Breaking changes to the schema require a major version bump
