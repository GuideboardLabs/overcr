# OverCR v2.6.0 — Execution Governance

## Purpose

This document defines the governance rules for sandbox execution. Every rule here is enforced by the software. Nothing is advisory — a violation means execution is blocked and an audit record is generated.

## Governance Layers

### Layer 1: Operator Approval
- Every execution requires an `approval_artifact` with `approved: true`
- The artifact must name the operator who approved
- No auto-approval path exists
- Approval is per-execution (not session-scoped)

### Layer 2: Command Allowlist
- Only 14 commands are allowed: ls, pwd, cat, head, tail, grep, find, echo, cp, mv, mkdir, rm, touch, diff
- Git is allowed only for `status` and `diff` sub-commands
- All other commands are blocked regardless of arguments
- Adding a command requires explicit operator action (code change)

### Layer 3: Argument Safety
- Shell metacharacters in any argument: blocked
- Command chaining operators (&&, ||, ;): blocked
- Pipes (|): blocked
- Redirections (>, >>, <): blocked
- Path traversal (../, ..\\): blocked
- Environment injection (PATH=, LD_PRELOAD): blocked

### Layer 4: Filesystem Boundary
- All writes are confined to the sandbox root directory
- Protected system paths (/etc/, /boot/, /sys/, /proc/, etc.) are permanently blocked
- Paths are resolved with `realpath()` to prevent symlink escapes
- Before any mutation, the target file is snapshotted for rollback

### Layer 5: Network Boundary
- All outbound network access is denied
- Network tool commands are blocked by name (curl, wget, ssh, nc, etc.)
- URL patterns in arguments are detected and blocked
- IP address patterns are detected and blocked
- Localhost access is denied
- Attempted network use generates audit records

### Layer 6: Execution Control
- `shell=False` in every subprocess call (argv array only)
- Timeout enforced at kernel level (SIGTERM)
- Exit code, stdout hash, and stderr hash recorded
- stdout/stderr stored to separate files (never inline in receipt)
- Execution elapsed time recorded to millisecond precision

### Layer 7: Rollback
- Pre-execution snapshots capture file content and metadata
- Snapshots are append-only and preserved indefinitely
- Rollback restores files to snapshot state
- Failed rollbacks generate audit records
- Rollback events are recorded in the receipt

### Layer 8: Audit Trail
- Every execution produces exactly one `ExecutionReceipt`
- Receipts are JSON files stored in `.overcr_receipts/`
- Receipts are never deleted
- Every receipt contains: operator identity, approval chain, full argv, output hashes, timing, modified paths, snapshot refs, governance flags
- Blocked executions still produce receipts (with `blocked_by_policy: true`)
- Failed executions still produce receipts

## What Is Never Allowed

- sudo, doas, pkexec, su — permanently blocked
- Package installation (apt, pip, npm, cargo, gem, go) — permanently blocked
- Shell interpreters (bash -c, sh -c, python -c, node -c) — permanently blocked
- Network tools (curl, wget, ssh, nc, telnet, ftp) — permanently blocked
- Privilege escalation (chmod, chown, setfacl) — permanently blocked
- Process control (kill, pkill, killall) — permanently blocked
- Filesystem mounts (mount, umount) — permanently blocked
- Firewall modification (iptables, nft, ufw) — permanently blocked
- Container runtime (docker, podman) — permanently blocked
- Cron/at scheduling (cron, crontab, at, batch) — permanently blocked
- Git commits, pushes, clones — permanently blocked

## Operator Requirements

The operator:
- Must provide an explicit approval artifact for every execution
- Is identified in the execution request and receipt
- Is responsible for reviewing execution receipts
- Must explicitly approve rollback operations
- Cannot delegate approval authority to automated systems

## Security Posture

The sandbox operates on a defense-in-depth model:
1. Policy layer blocks invalid commands before any process is spawned
2. Filesystem layer prevents writes outside the sandbox
3. Network layer prevents any outbound connection
4. Subprocess layer uses argv arrays (no shell interpolation)
5. Kernel timeout prevents runaway processes
6. Audit layer records everything that happened

Each layer operates independently — bypassing one does not bypass others.

## Remaining Attack Surfaces (Honest Assessment)

The following attack surfaces remain in v2.6.0:

1. **subprocess without container isolation**: The subprocess runs as the same UID as the OverCR process. A vulnerability in an allowed binary (e.g., `find` with -exec in older versions) could theoretically escalate. Mitigation: the argument scanner catches `-exec` patterns.

2. **No seccomp filtering**: The sandbox does not restrict syscalls at the kernel level. A compromised binary could make syscalls outside the intended scope. Mitigation: Firejail/Bubblewrap integration planned for v2.7.

3. **Filesystem race conditions**: There is a TOCTOU window between snapshot creation and mutation. An attacker with concurrent filesystem access could exploit this. Mitigation: the sandbox runs single-threaded and the window is extremely narrow.

4. **Argument parsing edge cases**: Different binaries parse arguments differently. `echo ">"` might be treated as text by echo but as redirect by a parser. Mitigation: the arg scanner rejects literal `>` tokens.

5. **Environment variable leakage**: The subprocess runs with `env={}` but the parent process environment is still available in `/proc/self/environ`. Mitigation: full kernel isolation (v2.7) addresses this.

These surfaces are acknowledged and tracked. None present an immediate path to unauthorized escalation in the current threat model, but they are not claimed to be eliminated.
