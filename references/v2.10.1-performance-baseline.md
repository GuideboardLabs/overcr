# OverCR v2.10.1 Performance Baseline

**Version:** v2.10.1  
**Purpose:** Measure deterministic latency for core v2.10.0 operations.  
**Governance:** Report-only. No network. No claims beyond measured environment.

## Operations Measured

| # | Operation | What it times |
|---|-----------|---------------|
| 1 | `workflow_execution` | Template register → load → validate |
| 2 | `packet_validation` | L1-L6 validate_packet on valid sample |
| 3 | `workflow_replay` | Replay validator on deterministic trace |
| 4 | `knowledge_index_search` | Keyword search on small index |
| 5 | `sandbox_policy_check` | Command policy validation (dry-run) |
| 6 | `release_manifest_gen` | Manifest.generate() call |

## Methodology

- **Sample workload:** Deterministic—same input every run
- **Warmup:** 3 warmup iterations per operation (excluded from stats)
- **Measurements:** 10 timed iterations per operation
- **Garbage collection:** `gc.collect()` between every measurement
- **Isolation:** All operations run in fresh temp directories
- **Output:** JSON with p50, p95, p99, avg, min, max per operation

## Running

```bash
python scripts/run_performance_baseline.py
```

Or programmatically:
```python
from validation.performance_baseline import PerformanceBaseline
baseline = PerformanceBaseline(warmup_samples=5, measurement_samples=50)
report = baseline.run()
print(baseline.to_report(report))
```

## Interpretation

These numbers measure **code path latency** in the current environment. They are NOT:
- Benchmark comparisons (no baseline from other versions)
- Production throughput estimates
- Claims about performance on other hardware/OS

Higher is not necessarily worse—the important metric is stability (low variance) and absence of pathological outliers (>10x median).

## Expected Values (WSL/Linux, Python 3.10+)

Typical ranges for a modern dev machine:
- Packet validation: <5ms
- Workflow execution: <50ms
- Remaining operations: <100ms

Values exceeding 500ms for any operation warrant investigation.
