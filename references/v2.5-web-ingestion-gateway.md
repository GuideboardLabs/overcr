# OverCR v2.5.0 — Controlled Web Ingestion Gateway

## Release Summary

v2.5.0 adds an operator-auditable single-page web ingestion gateway. An operator provides a URL and purpose; the gateway fetches exactly that page, validates it against governance rules, normalizes the content, scans for prompt injection risks, and optionally registers it as a provenance-aware knowledge source in the v2.4 knowledge layer.

This is **NOT** autonomous crawling. No recursive link following, no browser automation, no background fetch daemons. Every URL must be explicitly operator-provided with a stated purpose.

## Architecture Decisions

### 1. Operator-Initiated Only
Every fetch starts with a `URLRequest` that records the URL, the operator who requested it, the purpose, and the project scope. Nothing is auto-fetched. The request itself is the audit record of intent.

### 2. Scheme and Network Enforcement
`FetchGateway.validate_url()` blocks:
- Non-http/https schemes (file://, ftp://, data:, javascript:)
- Private IP addresses (10.x, 172.16.x, 192.168.x, 127.x, 169.254.x, 0.x)
- Localhost variants
- Redirects that resolve to private IPs

### 3. Mockable HTTP Client
The `FetchGateway` accepts an injectable `http_fetcher` callable. Tests inject a mock that returns controlled responses — the test suite requires NO external network. The real `requests`-based fetcher is only used in production/demo runs.

### 4. Content Normalization
`ContentNormalizer` converts HTML to clean markdown: preserves title, headings, canonical URL; strips script/style/nav/footer boilerplate; captures outbound links as metadata only (never follows them). Plain text is whitespace-normalized.

### 5. Robots.txt Advisory Check
`RobotsPolicy` can parse robots.txt rules but always records the result as advisory. If robots.txt is unavailable, status is "unknown" — policy uncertainty is always visible. The operator decides whether to proceed.

### 6. Prompt Injection Scanning
`PromptInjectionScanner` detects six categories of injection patterns: instruction override, system/operator override, secret exfiltration, authority claims, hidden injection (zero-width chars, HTML comments), and tool manipulation. Risks are FLAGGED but content is NEVER deleted. Risk levels: none, low, medium, high, critical.

### 7. Knowledge Layer Bridge
`WebSourceBuilder` converts a `FetchResult` into a v2.4 `SourceRegistry` record with full provenance chain, fetch metadata, robots policy result, and injection report. Default trust tier is "unknown" — escalated to "suspicious" if injection risk is critical/high.

### 8. Failure Records
Even failed fetches produce auditable failure records. The attempt — who, what URL, why, what went wrong — is preserved in `knowledge/reports/fetch_failure_*.json`.

## File Structure

```
web_ingestion/

├── prompt_injection_scanner.py      # Injection pattern detection
├── web_source_builder.py            # FetchResult → knowledge source bridge

    ├── url_request.schema.json      # URLRequest JSON schema
    └── web_source_record.schema.json # Web source record schema
```

## Governance Boundaries (Non-Negotiable)

1. No autonomous web crawling
2. No recursive link following
3. No browser automation
4. No login/authenticated scraping
5. No form submission
6. No background fetch daemon
7. No silent memory mutation
8. No trust auto-escalation (starts at "unknown", only operator can change)
9. No weakening L1-L6 validation
10. No bypassing provenance requirements
11. Fetched content is advisory — never canonical truth
12. Default trust tier for web sources is "unknown"

## Mock vs Real

The test suite injects a `mock_fetcher` that returns controlled HTML responses. **Zero external network calls** in tests. The real `requests`-based fetcher is used only in:
- `demo_web_ingestion_gateway.py` (interactive demo)
- Production workflow execution (operator-initiated)

## Future v2.6+ Extension Points

1. **PDF Extraction** — `ContentNormalizer` extended with PDF text extraction via pymupdf
2. **CryER Integration** — Web sources as inputs to CryER signal extraction workflows
3. **Workflow Integration** — `fetch_url` as a governed workflow node with approval gating
4. **Crawl Budget** — Operator-defined depth-1 crawling (follow links on same domain only)
5. **Browser Adapter** — Optional Playwright-based rendering for JS-heavy SPAs
6. **Rate Limiting** — Per-domain fetch rate limits to respect server load
7. **Content Diff** — Detect and track changes when re-fetching the same URL
