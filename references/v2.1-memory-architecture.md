# OverCR v2.1.0 — Semantic Memory Architecture

## 1. Overview

The semantic memory layer adds advisory context to the OverCR substrate without
compromising its filesystem-first canonical state model. Memory records are
**informative, not authoritative** — they can suggest, remind, and warn, but they
cannot override task state, mutate operational files, or bypass governance.

## 2. Design Principles

| Principle | Enforcement |
|-----------|-------------|
| Filesystem is canonical truth | MemoryManager reads/writes JSON files on disk; no in-memory cache |
| Memory is advisory | `canonical_state_refs` link to truth; consumer must verify |
| No autonomous learning loops | Promotion requires explicit rules; no model-output ingestion |
| No self-modifying prompts | Memory records are data, not prompt templates |
| No vector DB lock-in | Retrieval is keyword/tag-based; embeddings reserved for v2.2+ |
| No cloud dependencies | All storage is local filesystem; JSON/markdown friendly |
| Rejected memories persist | Rejected records stay on disk; never silently deleted |
| Stale memories are recoverable | `stale` → `active` transition is valid |
| Every mutation is audited | Append-only `index.jsonl` logs every create/status change |

## 3. Data Model

Each memory record is a JSON file at `memory/records/mem-XXXXXXXX.json`
conforming to `memory/schema/memory_record.schema.json`.

### 3.1 Core Fields

- **memory_id**: Unique identifier (`mem-` + 8 hex chars). Immutable.
- **source**: Who/what produced this memory (operator name, subagent, rule ID).
- **created_at / updated_at**: ISO 8601 timestamps. `created_at` is immutable.
- **confidence**: 0.0–1.0. Set by provenance rule, never auto-adjusted.
- **provenance**: Object with `type`, `rule`, and optional `operator_id`,
  `task_id`, `artifact_path`. Every memory has a mandatory provenance chain.
- **tags**: String array, min 1. Used for keyword retrieval.
- **project_scope**: String. Enables scoped retrieval per project/domain.
- **semantic_summary**: Human-readable text (1–4096 chars). Advisory context.

### 3.2 Reference Fields

- **supporting_artifacts**: `[{path, description}]` — filesystem paths to
  evidence. Advisory — does not grant access or mutation rights.
- **canonical_state_refs**: `[{path, field, as_of}]` — links to OverCR's
  authoritative filesystem state. The memory is **derived from** this state.
  If state changes, the memory may become stale.
- **contradiction_refs**: `[{conflicting_memory_id, conflict_type, detected_at, note}]` —
  links to other memories that contradict this one. Populated by
  MemoryConflictResolver.

### 3.3 Status Lifecycle

```
active ──→ stale ──→ active  (refresh cycle)
  │          │
  │          ├──→ superseded  (superseded_by field set)
  │          └──→ rejected    (terminal, record retained)
  │
  ├──→ superseded
  └──→ rejected
```

Terminal states (`rejected`, `superseded`) are immutable. Records in these
states remain on disk indefinitely.

## 4. Components

### 4.1 MemoryRecord (`memory/memory_record.py`)

Validated data model. Factory methods:
- `MemoryRecord.create(...)` — creates with auto-generated ID and timestamps
- `MemoryRecord.from_dict(data)` — reconstructs from stored JSON
- `MemoryRecord.from_json(json_str)` — reconstructs from JSON string

State machine enforcement:
- `update_status(new_status, reason)` — validates transitions
- `supersede(replacement_id)` — marks as superseded, sets superseded_by
- `add_contradiction(conflicting_id, conflict_type, note)` — adds contradiction ref

### 4.2 MemoryManager (`memory/memory_manager.py`)

Filesystem-backed CRUD. Methods:
- `create_memory(...)` — validates and persists a new record
- `load_memory(memory_id)` — reads from disk
- `search_memory(tags, project_scope, status, source, text_query)` — AND-combined filter
- `update_memory_status(memory_id, new_status, reason)` — state transitions
- `list_project_memory(project_scope, status)` — scoped listing
- `supersede_memory(old_id, new_id, reason)` — mark old as superseded

All mutations write to disk immediately. Appends to `index.jsonl` audit log.

### 4.3 MemoryPromoter (`memory/memory_promoter.py`)

Governed promotion from operational artifacts. Rules:
| Rule | Source | Confidence | Trigger |
|------|--------|-----------|---------|
| `task_completion_insight` | subagent_output | 0.7 | Completed task with summary |
| `operator_observation` | operator_direct | 0.9 | Explicit operator input |
| `filesystem_artifact_promotion` | filesystem_artifact | 0.6 | Filesystem artifact observation |
| `validation_lesson` | promotion_rule | 0.8 | L1–L6 validation failure |

Governance gates:
1. Rule must exist in `PROMOTION_RULES` — no ad hoc promotion
2. Confidence override may only **decrease**, never exceed rule ceiling
3. `operator_direct` provenance requires explicit `operator_id`

### 4.4 MemoryRetriever (`memory/memory_retriever.py`)

Keyword/tag retrieval with deterministic fallback cascade:
1. tags + project_scope + status=active → exact
2. Any tag match + project_scope + active → relaxed
3. project_scope + active → scope-only
4. project_scope + any status → broad
5. text_query across all → last resort
6. All active → widest

Additional methods:
- `retrieve_by_id(memory_id)` — single record lookup
- `retrieve_active_for_project(scope, tags)` — convenience
- `retrieve_with_state_refs(scope, canonical_path)` — filesystem-first path

### 4.5 MemoryConflictResolver (`memory/memory_conflict.py`)

Detection only — **no auto-resolution**. Methods:
- `detect_conflicts(project_scope)` — scans for contradictions
- `load_conflict(conflict_id)` — loads a review artifact
- `list_conflicts()` — lists all review artifacts

Detection strategies (v2.1, keyword-based):
1. Overlapping tags + conflicting summaries (negation detection)
2. Same canonical_state_ref pointing to different conclusions
3. Bidirectional contradiction_refs linking

Output: `conflicts/conflict-XXXXXXXX.json` review artifacts with
`resolution: null` — to be filled by operator.

## 5. Storage Layout

```
<overcr_root>/memory/
├── README.md
├── __init__.py
├── memory_record.py
├── memory_manager.py
├── memory_promoter.py
├── memory_retriever.py
├── memory_conflict.py
├── schema/
│   └── memory_record.schema.json
├── (runtime, created by MemoryManager)
│   ├── records/
│   │   └── mem-XXXXXXXX.json
│   ├── conflicts/
│   │   └── conflict-XXXXXXXX.json
│   └── index.jsonl
```

## 6. Obsidian Compatibility

Memory records are JSON files with human-readable fields. For Obsidian:
- `semantic_summary` maps to a note's body text
- `tags` maps to Obsidian tags
- `project_scope` maps to a folder or YAML frontmatter field
- `canonical_state_refs` links to other vault notes
- The `index.jsonl` can be rendered as a Dataview source

Future work: Markdown export of memory records with YAML frontmatter.

## 7. Relationship to Existing OverCR Architecture

### 7.1 Task Store

Memory records reference tasks via `provenance.task_id`. The task store
(`runtime/task_store.py`) remains the canonical source for task state.
Memory records about tasks are **advisory annotations**, not task state.

### 7.2 Audit Writer

The `index.jsonl` append-only log follows the same pattern as
`runtime/audit_writer.py`. Every create and status change produces an
index entry. This is separate from but analogous to the audit trail.

### 7.3 Validation (L1–L6)

Memory records do NOT participate in packet validation. They are not
part of the L1–L6 pipeline. They inform operator and subagent context
but are never used to validate or reject packets.

### 7.4 Governance Boundaries

The governance boundaries in `docs/GOVERNANCE_BOUNDARIES.md` are
unmodified. Memory does NOT:
- Bypass approval gates
- Override validation
- Mutate operational state
- Generate outbound actions
- Auto-promote from model output
