# OverCR v1.0.0 — Release Definition

## Date
2026-05-11 — Phase 1 of stabilization cycle.

## What v1.0.0 Means

v1.0.0 is the **first stable release** of the OverCR portable AI orchestration substrate.

It does NOT add features. It does NOT add models, subagents, packet types, or workloads.
It freezes the surface area of the substrate and certifies it as stable, documented,
and backward-guaranteed.

v1.0.0 is the version at which:

1. The substrate API (packet contracts, task lifecycle, workflow DAG model, governance
   boundaries) becomes **stable** — no breaking changes without a major version bump.
2. All existing tests (26 test modules, 26/26 pass) are **certified as release gates**.
3. The **Hermes reference runtime boundary** is codified as part of the substrate contract.
4. The **Open WebUI optional layer boundary** is codified as non-essential.
5. **Governance guarantees** (no autonomous outbound action, no autonomous filesystem
   mutation) are declared as invariant substrate properties.

v1.0.0 is the substrate saying: "what you see is what you get."
Everything inside the scope boundary below is supported. Everything outside is
explicitly NOT supported.

## Version Convention

- **v1.0.0** = stabilization release (this document)
- **v1.N.x** = patch releases (bug fixes, docs, hardening) on the v1.0 surface
- **v2.0.0** = next major release (requires scope revision and breaking change review)

## Out-of-Scope Declaration

The following are **explicitly NOT part of v1.0.0** and will not be added during
the stabilization cycle:

| Category | Examples |
|----------|----------|
| New subagents | No additional workers beyond CryER, PypER, CodER, KnowER |
| New packet types | No additions to the 27 registered packet types |
| New task states | 12-state machine is final for v1.x |
| New workflow primitives | DAG model is final for v1.x |
| New runtime bindings | No native Open WebUI integration, no new CLI adapters |
| New workloads | CRM, outreach, research, coding are EXAMPLE applications, not substrate identity |
| Multi-node / distributed | Single-instance substrate only |
| Web service layer | No HTTP server, no REST API, no webhooks (keep webhook skill separate) |
| Authentication / authorization | No user model, no RBAC, no API keys managed by substrate |
| Encryption at rest | No built-in encryption of task records or audit logs |
| Migration tooling | No automated migration from v0.x formats |
| Packaging | Release packaging (`scripts/package_release.sh`) is tooling, not substrate behavior |

These items remain on the horizon for future releases or are intentionally
permanent exclusions.

## Hermes Reference Runtime Boundary

Hermes is the **primary reference runtime**. v1.0.0 codifies this boundary:

- OverCR defines WHAT to request (model, domain, prompt, governance constraints).
- Hermes executes HOW (provider API call, local inference, timeout, streaming).
- OverCR NEVER executes models, connects to providers, or manages API keys.
- Hermes NEVER bypasses governance, skips validation, or auto-approves tasks.

This boundary is structural and invariant. See `docs/HERMES_REFERENCE_RUNTIME.md`
for the full contract.

---

## Open WebUI Optional Layer Boundary

Open WebUI is an **optional visual oversight layer**. v1.0.0 codifies:

- Open WebUI is NOT required for substrate operation.
- Open WebUI has NO native integration with OverCR state or governance.
- If Open WebUI is present, it operates as a peer consumer of substrate output,
  not as a controller.
- Substrate state remains filesystem-first; Open WebUI reads but does not write.

---

## No Autonomous Outbound Action Guarantee

**In v1.0.0, no code path in the substrate initiates outbound network requests,
sends email, makes API calls, or writes to external systems.**

This is enforced by:
- `ApprovalGate.should_block_outbound()` — hard block by default
- `WorkflowPolicy._check_packet_content_safety()` — pattern scanning for network/shell
- Worker contract — stdout-only JSON, no socket/file access
- Runtime boundary — OverCR has no network stack

---

## No Autonomous Filesystem Mutation Guarantee

**In v1.0.0, no inference path writes to the filesystem outside of its
explicit output directory (`tasks/`, `logs/`, `audits/`).**

This is enforced by:
- `TaskStore` owns all task record writes
- `AuditWriter` owns all audit log writes (append-only)
- Worker stdout capture — workers cannot write files
- Workers that attempt filesystem access fail the worker contract
