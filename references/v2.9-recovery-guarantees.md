# OverCR v2.9.0 — Recovery Guarantees

## Guaranteed Recovery Paths

### 1. Cold-Start from Fresh Clone

A `git clone` followed by `cd` into the repository is a valid starting state.
Required directories, schemas, and workflow templates are self-contained
within the repository. Runtime state (audit traces, task records) is
generated on first execution; none is required to bootstrap.

**Guarantee**: First workflow execution succeeds from a fresh clone with
no additional initialization steps.

### 2. Deterministic Replay

Two executions of the same workflow template with identical inputs produce
the same node execution order and the same sequence of audit entry types.
Timestamps differ (wall clock), but structure and ordering are identical.

**Guarantee**: `replay_workflow()` reconstructs the exact same execution
trace as the original run.

### 3. Audit Trace Reconstruction

All audit entries are tagged with `run_id`, `workflow_id`, `entry_type`,
and `timestamp`. Entries can be grouped by run to reconstruct the full
execution timeline. Node states, approvals, validations, fallbacks, and
rollbacks are all recorded.

**Guarantee**: Any completed workflow execution can be fully reconstructed
from its audit trace.

### 4. Partial Artifact Tolerance

Missing trace files are detectable (file count mismatch vs. expected).
Remaining traces can be loaded and parsed independently. A missing trace
does not corrupt other traces.

**Guarantee**: The system detects missing artifacts rather than silently
ignoring them. Remaining artifacts remain fully parseable.

### 5. Rollback Restoration

RollbackSnapshots capture pre-execution state before sandbox operations.
Snapshots are serialized to JSON and can be loaded, verified by snapshot_id,
and used to reconstruct the pre-execution environment.

**Guarantee**: Snapshot save/load round-trip preserves snapshot_id and
all captured state fields.

## What Is NOT Guaranteed

### 1. Timestamp Exactness
Timestamps in separate runs WILL differ. Replay matches structure, not
wall-clock timing.

### 2. Cross-Run Output Identity
Deterministic packets contain run-specific UUIDs and timestamps. Two
runs produce equivalent but not byte-identical outputs.

### 3. Concurrent Workflow Safety
Multiple workflows running simultaneously on the same filesystem may
interleave audit trace writes. The system is designed for sequential
execution; parallel safety is not verified by v2.9.0.

### 4. Network-Isolated Recovery
Recovery scenarios assume a writable local filesystem with Python
available. Air-gapped or network-restricted environments where
required packages are not installed are not tested.

### 5. Corrupted Trace Recovery
A partially written or truncated trace file (from a crash during
execution) may not be fully reconstructable. The system appends
atomically per-line, but partial final lines are possible.

## Recovery Boundary

Recovery verification simulates scenarios using `tempfile.TemporaryDirectory`.
Production state is NEVER touched during recovery simulation. This
guarantees that recovery testing cannot itself be a source of state
corruption.
