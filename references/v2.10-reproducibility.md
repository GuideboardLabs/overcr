# OverCR v2.10.0 — Reproducibility Guarantees

## Determinism Claims

### Workflow Execution
- Same template + same initial_input → same node order (verified by ReplayValidator)
- Same template + same initial_input → same audit structure (entry types in order)
- Same template + same initial_input → same final workflow_state

### Schema Registry
- discover_all() is idempotent (same call, same result)
- list_schemas() is idempotent
- Schema completeness checks are stable

### Release Manifest
- Package versions are stable across generate() calls
- Schema versions are stable
- Workflow versions are stable
- Backend information is stable (timestamps only vary)

### Execution Receipts
- to_dict() is idempotent (same object, same serialization)
- Round-trip (to_dict → from_dict) preserves identity fields

### Test Manifest
- Count and categories are verified
- Every entry has required fields (name, module, category)

## Non-Deterministic Elements (documented)

- Timestamps (generated_at, timestamp fields) vary by nature
- Git describe output varies with commit history
- Archive size may vary slightly with compression
- Environment detection reflects current state (deps, backends)

## Reproducibility Verification

Run `scripts/check_replay_consistency.py` to verify:
- All 5 workflow templates replay deterministically
- Audit reconstruction is temporally ordered
- Branch traces are internally consistent
- Receipts are round-trip safe

Run `release/ReproducibilityChecker.check_all()` for full report.
