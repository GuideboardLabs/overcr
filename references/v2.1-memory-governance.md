# OverCR v2.1.0 вҖ” Semantic Memory Governance

## 1. Purpose

This document defines the governance constraints for the OverCR semantic memory
layer. These constraints are **enforced in code**, not advisory.

## 2. Core Governance Rules

### Rule 1: Memory Is Advisory

Semantic memory records INFORM operational decisions but do NOT author them.

| What memory CAN do | What memory CANNOT do |
|-------------------|----------------------|
| Provide context to operators | Override task state on disk |
| Suggest relevant past patterns | Bypass approval gates |
| Flag contradictions for review | Auto-resolve contradictions |
| Link to canonical state refs | Mutate canonical state files |
| Add confidence signals | Substitute for validation |

**Enforcement**: No code path in MemoryManager, MemoryPromoter,
MemoryRetriever, or MemoryConflictResolver writes to anything outside
`memory/records/`, `memory/conflicts/`, or `memory/index.jsonl`.

### Rule 2: No Autonomous Promotion

Memory promotion requires an **explicit rule** from `PROMOTION_RULES`.

| Allowed | Forbidden |
|---------|-----------|
| `MemoryPromoter.promote(rule_name=..., ...)` | Reading model output directly into memory |
| Rules with defined source, confidence, scope | Inferring memory from chat history |
| Operator-driven `operator_observation` | Autonomous learning loops |
| `validation_lesson` from L1вҖ“L6 failures | Self-modifying prompt templates |

**Enforcement**: `PromotionError` raised for unknown rules. Confidence
override can only DECREASE, never exceed rule ceiling. `operator_direct`
requires `operator_id`.

### Rule 3: Provenance Is Mandatory and Non-Auto-Filled

Every memory record has a `provenance` object with required fields:
- `type`: One of `operator_direct`, `promotion_rule`, `subagent_output`, `filesystem_artifact`
- `rule`: Identifier of the promotion rule or explicit creation method

Provenance is set at creation time and **never auto-generated** from
model context or inference output.

**Enforcement**: `MemoryRecord._validate_data()` rejects records without
provenance. The `create` factory method requires `provenance_type` and
`provenance_rule` arguments.

### Rule 4: Rejected Memories Never Silently Disappear

When a memory record transitions to `rejected` status:

1. The record remains on disk in `memory/records/`
2. The status field is set to `rejected`
3. The `updated_at` timestamp is updated
4. An index entry is appended (action: `status_change:active->rejected`)
5. The record is findable via `search_memory(status="rejected")`

There is no delete operation. Rejected memories are excluded from active
retrieval by default but remain auditable and recoverable.

**Enforcement**: `MemoryManager` has no `delete_memory()` method. The status
state machine makes `rejected` a terminal state with no outgoing transitions.

### Rule 5: Stale Memories Remain Recoverable

The `stale` status indicates a memory may be outdated, not that it is
discarded. Transition `stale` вҶ’ `active` is valid (refresh cycle).

When marking stale:
- `stale_reason` must be provided (context for the operator)
- `updated_at` is updated
- The record remains on disk and searchable

**Enforcement**: `update_status("stale", reason=...)` requires reason for
auditability. The state machine allows `stale` вҶ’ `active` transitions.

### Rule 6: Memory Cannot Override Task Truth

Memory records reference tasks via `provenance.task_id`, but:
- The task store (`runtime/task_store.py`) is authoritative for task state
- Memory summaries about tasks are annotations, not corrections
- `canonical_state_refs` point to authoritative files вҖ” memory is *derived*

If a memory's `canonical_state_refs` point to a file that has since changed,
the memory may be stale. The consumer is responsible for verifying refs
against current filesystem state.

**Enforcement**: `MemoryRetriever.retrieve_with_state_refs()` is provided
for filesystem-first retrieval. No code path writes to task files, audit
logs, or any operational state outside the `memory/` directory.

### Rule 7: Retrieval Must Be Auditable

Every search and retrieval operation is deterministic and filesystem-based:

1. `search_memory()` reads all `mem-*.json` files from disk (no cache)
2. Results are sorted by `updated_at` descending (deterministic order)
3. Deterministic fallback cascade is well-defined (6 levels)
4. `retrieve_with_state_refs()` explicitly links to canonical state

No retrieval depends on vector similarity, embedding scores, or probabilistic
ranking. Any future embedding-based retrieval MUST preserve the deterministic
path as a fallback.

**Enforcement**: `MemoryRetriever` provides `deterministic_fallback=True`
(default) with a documented, deterministic cascade. Setting
`deterministic_fallback=False` skips the cascade but still uses disk-based
structured search вҖ” no probabilistic path exists.

### Rule 8: No Cloud Dependencies

All storage is local filesystem. No network calls, no API calls, no
database connections.

| Component | Storage | Network |
|-----------|---------|---------|
| MemoryManager | `<root>/memory/records/` | None |
| MemoryPromoter | Delegates to MemoryManager | None |
| MemoryRetriever | Reads from MemoryManager | None |
| MemoryConflictResolver | `<root>/memory/conflicts/` | None |

**Enforcement**: No `import requests`, `import httpx`, `import psycopg2`,
or any network library in any memory module. Verified by code inspection.

## 4. Promotion Governance

### 4.1 Promotion Rules Are Code-Defined

Rules are defined in `PROMOTION_RULES` dict in `memory_promoter.py`.
Adding a new rule requires:
1. Adding the rule definition to `PROMOTION_RULES`
2. Testing that the rule produces valid MemoryRecord fields
3. Updating this governance document

No rule can be created at runtime. No rule can be loaded from model output.

### 4.2 Confidence Ceiling

Each rule has a default confidence. Overrides may only **decrease** confidence,
never increase it above the rule's ceiling.

| Rule | Ceiling | Rationale |
|------|---------|-----------|
| operator_observation | 0.9 | Human-sourced, high trust but not canonical |
| validation_lesson | 0.8 | Derived from L1вҖ“L6 failures, high signal |
| task_completion_insight | 0.7 | Model output, advisory only |
| filesystem_artifact_promotion | 0.6 | Filesystem state is canonical; memory is derived |

### 4.3 Provenance Enforcement

| Type | Required Fields |
|------|----------------|
| `operator_direct` | `operator_id` |
| `promotion_rule` | `rule` (auto-filled) |
| `subagent_output` | `task_id` |
| `filesystem_artifact` | `artifact_path` |

## 5. Conflict Resolution Governance

### 5.1 Detection Only

`MemoryConflictResolver.detect_conflicts()` identifies potential
contradictions and creates review artifacts. It does NOT:

- Auto-resolve contradictions
- Modify memory record status
- Delete records
- Choose which record is "correct"

### 5.2 Review Artifacts

Conflict detection produces `conflict-XXXXXXXX.json` files in
`memory/conflicts/`. Each contains:

```json
{
  "conflict_id": "conflict-XXXXXXXX",
  "detected_at": "ISO-8601",
  "conflict_type": "factual_contradiction|temporal_contradiction|scope_overlap",
  "records": [
    {"memory_id": "...", "source": "...", "semantic_summary": "...", ...},
    {"memory_id": "...", "source": "...", "semantic_summary": "...", ...}
  ],
  "analysis": "Human-readable explanation",
  "resolution": null
}
```

The `resolution` field starts as `null`. An operator fills it in after review.
No code path sets it automatically.

### 5.3 Contradiction Refs

When a conflict is detected, `contradiction_refs` are added to BOTH
records bidirectionally. This creates a traceable link that persists even
if one record is later rejected or superseded.

## 6. Future Governance (v2.2+)

The following are **NOT** implemented in v2.1 and must go through the same
governance review before addition:

- Embedding-based retrieval (requires: deterministic fallback preservation)
- RAG pipeline (requires: advisory-only constraint preservation)
- Autonomous memory synthesis (requires: explicit rule governance)
- Web ingestion (requires: provenance chain extension)
- PLUR adapter (requires: interface compatibility review)
