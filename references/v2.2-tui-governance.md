# OverCR v2.2.0 — TUI Governance Constraints

## Core Invariant

The TUI is an **observatory**, not a **cockpit**. It renders operational
state for human awareness and decision support. It does not create
operational truth.

## Constraint 1: TUI Cannot Bypass Runtime Policy

The TUI cannot directly modify task state, skip approval gates, or
override governance decisions. All state mutations go through the
OverCR runtime, which enforces:

- L1-L6 validation on all packets
- Approval gates on PypER/outbound tasks
- State machine enforcement (TaskStore.VALID_TRANSITIONS)
- Audit logging on every state transition

**Enforcement:** TUI views are read-only. The only "action" the TUI
produces is `ApprovalAction` — a data object that must be routed
through `OverCRRuntime.process_approval()` to have any effect.

## Constraint 2: TUI Actions Must Route Through Governance

`ApprovalQueue.propose_approval()` returns an `ApprovalAction` dataclass.
It does NOT call `OverCRRuntime.process_approval()`. The caller must:

1. Review the proposed action
2. Route it through `OverCRRuntime` (which enforces `ApprovalGate`)
3. Accept the result (which may be different from what was proposed)

This ensures that the TUI cannot short-circuit governance, even if the
operator "clicks approve" in the interface.

## Constraint 3: TUI Reflects Operational Truth

All TUI views read from canonical filesystem state:
- `orchestration/tasks/` for task records
- `runtime/audit.jsonl` for audit entries
- `memory/index.jsonl` for memory state
- `subagents/` for worker availability

If the filesystem says state X, the TUI shows state X. The TUI never
caches, invents, or interpolates state.

## Constraint 4: All Operator Actions Must Remain Auditable

Every `ApprovalAction` carries:
- `task_id` — which task is being acted on
- `action` — "approve" or "reject"
- `rationale` — human-readable reason
- `operator` — who is making the decision

When the caller routes this through `OverCRRuntime`, it triggers
`AuditWriter.approval_action()`, which appends an immutable audit
entry. The operator's decision is permanently recorded.

## Constraint 5: Deterministic Fallback Required

Every view has both `render()` (rich) and `render_plain()` (plain text).
The plain text fallback produces semantically equivalent output without
rich markup. This enables:
- Testing without a terminal
- Log capture and comparison
- Diff-based regression testing
- Operation on constrained terminals

## Constraint 6: No Auto-Approval

The `ApprovalQueue` has NO method that automatically approves tasks.
It has:
- `render_queue()` — displays pending approvals
- `render_detail()` — shows rationale for a specific approval
- `propose_approval()` — creates an ApprovalAction (data, not execution)
- `propose_rejection()` — creates an ApprovalAction (data, not execution)

There is no `auto_approve()`, `approve_all()`, or any method that
executes approval without explicit operator decision.

## Constraint 7: Resilient to Partial Failure

If any subsystem is missing or malformed:
- Missing task files → "No tasks found" (not a crash)
- Missing audit log → "No entries matching filters" (not a crash)
- Missing memory index → Memory summary shows zeros (not a crash)
- Malformed JSON → Skipped with graceful degradation (not a crash)

The TUI never raises unhandled exceptions for missing data. It shows
what exists and notes what's absent.

## Constraint 8: No Mutable State in Views

TUI view classes are stateless renderers. They hold:
- `root` — path to OverCR core directory (immutable)
- `console` — Rich Console instance (immutable)
- `badge` — StatusBadge instance (stateless)

They do NOT hold:
- Task state caches
- Audit log caches
- Memory caches
- Worker registries

Every `render()` call reads fresh data from disk. This prevents stale
views from showing outdated state.

## Allowed Operations

| Operation | Allowed | Mechanism |
|---|---|---|
| Read task records | Yes | `TaskView._load_tasks()` reads from filesystem |
| Read audit log | Yes | `AuditView._read_audit_log()` reads from filesystem |
| Render formatted views | Yes | All `render()` methods |
| Propose approval action | Yes | `ApprovalQueue.propose_approval()` returns data |
| Propose rejection action | Yes | `ApprovalQueue.propose_rejection()` returns data |
| **Execute approval** | **No** | Must go through `OverCRRuntime` |
| **Bypass governance** | **No** | TUI has no mutable state access |
| **Auto-approve tasks** | **No** | No such method exists |
| **Modify task state** | **No** | TUI views are read-only |
| **Write to filesystem** | **No** | TUI views never write to runtime state |

## Testing Governance

The test suite (`tests/test_tui_views.py`) includes explicit tests for:
1. No auto-approval — verify `ApprovalQueue` has no auto-approve methods
2. ApprovalAction is data — verify it has no side effects
3. Deterministic rendering — same input produces same output
4. Degraded fallback — missing data produces graceful output, not crashes
