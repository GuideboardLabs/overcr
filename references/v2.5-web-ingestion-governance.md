# OverCR v2.5.0 — Web Ingestion Governance

## Purpose

This document defines the governance rules that every web fetch operation must satisfy. These rules are encoded in the software (validation, blocking, scanning) — this document serves as the human-readable specification.

## Fetch Governance Layers

### Layer 1: URL Validation
Every URL must pass validation before any network request:
- Scheme must be `http` or `https` — all others blocked
- Hostname must be present and resolvable
- Private IPs and localhost are blocked by default
- File://, ftp://, data:, javascript: schemes are permanently blocked

### Layer 2: Network Boundary
Before connecting:
- The gateway checks that the resolved address is not in private IP ranges
- Redirects are followed (if `follow_redirects=True`) but each redirect target is re-validated
- Redirects to private IPs are blocked mid-chain
- No DNS rebinding attacks possible — IP is checked post-resolution

### Layer 3: Response Safety
After receiving a response:
- Content-Type is validated against `allowed_content_types`
- Response size is capped at `max_bytes` (default 512KB)
- Truncation is noted in metadata but content beyond the cap is discarded
- Timeout is enforced at the HTTP level

### Layer 4: Robots.txt Advisory
- robots.txt for the target domain is checked where practical
- Result is recorded as: allowed, disallowed, unknown, or error
- robots.txt is ADVISORY — the operator decides whether to proceed
- Policy uncertainty ("unknown") is always visible in the audit record
- Never hidden or silently ignored

### Layer 5: Prompt Injection Scanning
All fetched content is scanned for:
- Instruction override patterns
- System/operator override attempts
- Secret exfiltration requests
- False authority claims
- Hidden injection via zero-width characters or HTML comments
- Model/tool manipulation language

Risk levels: none, low, medium, high, critical.
Content is NEVER deleted based on scan results — only flagged.
The operator reviews and decides.

### Layer 6: Trust Tier Assignment
- Default trust tier for all web sources is "unknown"
- If injection risk is critical or high, default escalates to "suspicious"
- Only `SourceRegistry.set_trust_level()` with operator identity can change tier
- No automatic escalation path exists beyond initial classification
- Web sources CANNOT become canonical truth automatically

## What Is Never Allowed

- Autonomous web crawling (no spidering, no recursive fetching)
- Browser automation (no Selenium, Playwright, Puppeteer)
- Authenticated scraping (no cookies, no auth headers, no login)
- Form submission (no POST, no data upload)
- JavaScript execution (no rendering engine)
- Background fetch daemon (every fetch is operator-initiated)
- Silent memory mutation (no auto-promotion from web content)
- Trust auto-escalation (operator must explicitly change trust tiers)
- Hidden provenance rewriting (append-only JSONL tracker)

## Operator Requirements

The operator:
- Provides the URL explicitly — no search-to-fetch automation
- States the purpose — recorded in provenance
- Reviews injection scan results before using content in workflows
- Assigns trust tiers explicitly (the system defaults to "unknown"/"suspicious")
- Is identified in every fetch request and provenance entry
- Is responsible for respecting robots.txt policy decisions

## Integration With Knowledge Layer

Web sources integrate with v2.4 knowledge layer through `WebSourceBuilder`:
- Registered as `SourceRegistry` records with type "website"
- Full provenance chain: fetch → normalize → scan → register
- Citation tracking works normally (web sources can be cited in research packets)
- Contradiction detection works across web and document sources
- Staleness applies — web sources can go stale like any other source

## Audit Trail Requirements

Every web fetch records:
- URL requested (original and final after redirects)
- Operator identity and purpose
- Fetch timestamp
- Response code and content type
- Response size and elapsed time
- Redirect chain (every hop)
- Robots.txt policy result
- Prompt injection risk level and all flagged patterns
- Content hash (SHA-256)
- Normalization method applied
- Registration source_id (if registered)
