# OverCR v2.7.0 — Kernel Isolation Sandbox Backend

## Release Summary

v2.7.0 adds optional kernel-level isolation backends to the v2.6 execution sandbox. The existing policy layer (CommandPolicy, NetworkGuard, FilesystemGuard) runs unchanged — backends are defense-in-depth underneath the policy.

Three backends are available: `local` (always available, no kernel features), `bubblewrap` (optional, requires bwrap), `firejail` (optional, requires firejail). None are installed or required. The `BackendSelector` picks the strongest available backend and audibly records every fallback decision.

This release is about hardening — not expanding autonomy.

## Architecture Decisions

### 1. Policy-First, Backend-Second
CommandPolicy, NetworkGuard, and FilesystemGuard run BEFORE backend selection. The allowlist, metachar detection, and approval gate are unchanged from v2.6. Backends only handle HOW a pre-approved command executes — never WHAT is allowed to execute.

### 2. Abstraction Interface
`SandboxBackend` defines the contract: `available()`, `build_command()`, `execute()`, `describe_isolation()`, and capability queries (`supports_network_block()`, etc.). Any backend implementing this interface can be plugged in — no changes to `SandboxRunner` needed.

### 3. Never Shell-Based
Every backend's `build_command()` returns a `list[str]` — never a string. Every `execute()` calls `subprocess.run(shell=False)`. No shell interpolation, no command string construction, no injection surface.

### 4. Auditable Fallback
`BackendSelector.select()` returns both the backend AND selection metadata. The metadata records: which backend was selected, which were attempted, whether fallback occurred, and why. This metadata is embedded in every receipt's `backend_selected` audit entry and the receipt's top-level fields.

### 5. Availability Detection (Not Installation)
Backends check availability at runtime via `shutil.which()`. No attempt is made to install missing tools. Absence is not a failure — it results in an audited fallback to local.

### 6. IsolationProfile
Declarative configuration serialized into every receipt. Defines what the operator intended (network off, readonly paths, resource caps) — the receipt proves whether the backend honored it.

## Backend Comparison

| Capability | Local | Bubblewrap | Firejail |
|---|---|---|---|
| Kernel namespace isolation | — | ✓ (unshare) | ✓ (profiles) |
| Network block | Policy only | --unshare-net | --net=none |
| Readonly mounts | Policy only | --ro-bind | --read-only |
| Private /tmp | — | --tmpfs | (via profile) |
| No /proc | — | Yes | (via profile) |
| Minimal /dev | — | Yes | (via profile) |
| Resource limits | Timeout only | Timeout only | --timeout, --rlimit |
| Seccomp | — | — | Default |
| Always available | ✓ | — | — |

## Governance Boundaries

1. Optional backend absence is not a failure — it's a documented fallback to local
2. Silent fallback is a failure — every fallback is audited
3. Stronger backend must never weaken command policy
4. Backend selection cannot grant extra command authority
5. Kernel isolation is defense-in-depth, not replacement for policy
6. No auto-install of missing backends

## Future v2.8+ Extension Points

1. **Seccomp profiles** — custom seccomp allowlists per command type for local backend
2. **User namespace** — unprivileged user namespaces for all backends
3. **cgroup v2 limits** — CPU/memory process limits via cgroupfs
4. **Podman backend** — optional rootless container isolation
5. **Profile templates** — predefined isolation profiles for common command categories
6. **TUI backend dashboard** — live display of which backend is active with isolation strength visualization
