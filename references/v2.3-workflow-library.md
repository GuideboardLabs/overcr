# OverCR v2.3.0 — Workflow Library

## Release Summary

v2.3.0 introduces a governed library of reusable workflow templates. These templates define operational choreography — sequences of bounded, deterministic, auditable, and replayable operations that pass through existing OverCR validation (L1-L6), approval gates, and audit boundaries.

This is **NOT** an autonomous agent swarm release. No agent has sovereignty over another. Every workflow node, edge, approval pause, rollback event, and fallback activation is recorded in append-only audit traces.

## Architecture Decisions

### 1. Filesystem-First, JSON-Native Templates
Templates are `.json` files stored in `workflow_library/templates/`. No database. No runtime compilation. A template can be copied, diffed, versioned, and replayed from any machine with an OverCR runtime.

### 2. Template Schema Validation
Every template must conform to `workflow_library/schema/workflow_template.schema.json`. The registry enforces this on registration and the executor enforces it before execution. Malformed, missing, or recursive templates are rejected before any node executes.

### 3. Context Isolation
Each workflow execution gets an isolated `WorkflowContext`. No shared mutable state. Every approval, node state change, and timing measurement is recorded inside the context's audit_entries list — which is append-only and never pruned.

### 4. Bridge to Existing Runtime
The `WorkflowExecutor` bridges workflow_library templates with the existing OverCR runtime (`workflow_runner.py`, `workflow_policy.py`, `validate_packet.py`). It does not duplicate validation logic — it routes through the same L1-L6 validator that all OverCR packets use.

### 5. Bounded Execution
Workflows have timeout_seconds. Every node has its own timeout. If a workflow exceeds its total timeout, it stops. If a node exceeds its timeout, the deterministic fallback policy determines the next action (stop, skip, or fallback).

### 6. Approval Gates Are Mandatory, Not Advisory
Nodes listed in `approval_points` cannot execute without operator approval. There is no auto-continue path. The executor pauses and records the pause. Only an explicit operator approval advances the workflow.

## Workflow Templates (v2.3.0)

| Workflow ID | Name | Description |
|---|---|---|
| `claim_review` | Claim Review | KnowER verifies a claim, checks contradictions, operator approves, final report generated |
| `recon_brief` | Recon Briefing | CryER extracts signals, KnowER analyzes sources, confidence scored, briefing packet produced |
| `coder_patch_review` | CodER Patch Review | CodER plans patch, risk assessment, rollback planning, operator approval, proposal generated |
| `execution_plan_review` | Execution Plan Review | PypER plans execution, safety review, refusal check, simulated run, receipt produced |
| `release_freeze` | Release Freeze | Validation checks, audit summary, RC report, freeze artifact, release packet produced |

## Governance Boundaries (Non-Negotiable)

1. Workflows may only route through OverCRRuntime — never direct subagent-to-subagent
2. Workflows cannot invoke shell execution directly
3. Workflows cannot mutate filesystem state without approval
4. Workflows cannot self-spawn recursively
5. Workflows cannot bypass `approval_required` fields
6. Workflows cannot hide failed nodes
7. Workflows cannot discard audit traces
8. Workflows cannot exceed declared timeout_seconds

## Audit Requirements

Every workflow execution records:
- `workflow_id` — which template
- `workflow_version` — which version of the template
- `node_execution_order` — which nodes ran in what order
- `approval_pauses` — every pause for operator approval
- `rollback_events` — every rollback and why
- `validation_results` — L1-L6 validation output per node
- `deterministic_fallback_activations` — when fallback was used
- `elapsed_timing` — per-node and total workflow timing

## What Was NOT Implemented (by Design)

- No live graph rendering (future Textual TUI integration point)
- No autonomous scheduling (future v2.4+)
- No cron execution
- No multi-user orchestration
- No distributed execution
- No cloud workflow storage
- No browser automation
- No self-modifying workflows
- No background daemons
- No uncontrolled recursion

## Future v2.4+ Extension Points

1. **Textual TUI Integration** — Live workflow graph rendering in the terminal, operator dashboards with pause/resume/reject buttons, audit trace viewer
2. **Workflow Composability** — Nested workflows (sub-workflows as nodes), conditional branching based on node output, parameterized templates
3. **Approval Queue** — Multi-workflow approval dashboard, operator role management, approval timeouts with auto-escalation
4. **Live Inference Mode** — Wire workflow nodes to actual Hermes inference calls (currently deterministic-only), model selection per node, confidence thresholds
5. **Template Versioning and Migration** — Detect template version changes on replay, migration paths for old templates
6. **Visual Editor** — JSON schema-driven form-based editor, DAG visual drag-and-drop, template validation preview
