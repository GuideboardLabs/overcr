# OverCR v2.4.0 — Provenance Governance

## Purpose

This document defines the governance rules for provenance tracking in the knowledge subsystem. Provenance is the foundational pillar of audit integrity — every piece of information must trace back to its origin through an unbroken, append-only chain of evidence.

## Provenance Principles

### 1. Append-Only Immutability
Provenance entries are written as JSONL (one entry per line) to tracker files. Once written, an entry is **never modified or deleted**. This is enforced by the file format — append-only is the only operation.

### 2. Full Lineage Tracking
Every source carries:
- **Origin**: where it came from (URL, path, operator upload, citation)
- **Ingestion path**: how it entered the system (ingest_markdown, ingest_json, ingest_text)
- **Transformation chain**: every normalization, classification, re-tagging
- **Workflow usage**: which workflow executions cited it
- **Citation references**: which research packets reference it
- **Contradiction lineage**: which other sources it conflicts with

### 3. Content Integrity Verification
Every ingested document gets a SHA-256 hash. The hash is stored in the source record. Duplicate content is detected by hash comparison at registration time. Content drift (re-ingestion of updated versions) is tracked through provenance.

### 4. Trust Tier Enforcement
Trust tiers are operator-controlled. The rules:
- `verified` requires explicit operator confirmation
- `reputable` requires operator attestation of source reliability
- `unknown` is the default for all new sources
- `suspicious` is operator-assigned when issues are found
- `rejected` is operator-assigned and means "do not use"
- **No automatic escalation exists** — the classifier never changes trust tiers
- **No model output can change trust tiers** — only `set_trust_level()` with operator identity

### 5. Contradiction Preservation
When contradictions are detected between sources:
- Both sides are preserved in full
- Both source IDs and trust tiers are recorded
- Severity is classified (direct, partial, contextual)
- Resolution status is set to "unresolved"
- **No automatic resolution occurs**
- **No source is hidden because it contradicts another**
- The operator must explicitly acknowledge, investigate, or dismiss

### 6. Source Lifecycle
Sources have three statuses:
- `active`: currently usable
- `stale`: may be outdated (flagged by staleness heuristic, operator review)
- `archived`: no longer actively used but fully retained

Sources are **never deleted**. They remain queryable at any status. The content files and metadata records persist indefinitely.

### 7. Citation Integrity
Every research packet cites its sources. Every citation updates the source's citation count. This creates a usage graph: which sources are most relied upon, which are under-used, and whether trust tiers align with actual usage patterns.

### 8. Replay Compatibility
Given the same set of source records and documents, the knowledge index, tracker files, and research packets are fully reproducible. This enables audit verification: replay the knowledge subsystem from a point-in-time snapshot and verify that the same packets would be produced.

## Audit Requirements

Every provenance operation records:
- Timestamp (UTC ISO 8601)
- Operation type (origin_recorded, ingestion_recorded, transformation_recorded, etc.)
- Operator identity
- Affected source IDs
- Details specific to the operation

These entries are the audit trail. They are the only source of truth for what happened to a source, when, and by whom. They cannot be fabricated retroactively because they are append-only and timestamped.

## Operator Responsibilities

The operator:
- Assigns trust tiers (the only path to change them)
- Reviews contradiction reports (the only path to resolve)
- Decides staleness (the classifier flags, the operator confirms)
- Is identified in every trust tier change and contradiction resolution

## Integration With Existing Governance

The knowledge subsystem operates within the existing OverCR governance framework:
- All research packets pass through L1-L6 validation
- All packets route through OverCRRuntime (never direct subagent-to-subagent)
- Approval gates apply to contradiction summaries (approval_required=True)
- Audit traces are written to runtime audit alongside workflow traces
- The knowledge subsystem cannot bypass any existing governance rule
